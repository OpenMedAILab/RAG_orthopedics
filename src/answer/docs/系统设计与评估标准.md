## 1. 系统概述

### 1.1 项目目标

本系统用于评估和比较 RAG（检索增强生成）与非 RAG 模式下大语言模型在医学问答任务中的表现。通过对比不同模型在有/无知识库检索支持下的回答质量，提供全面的评估和分析。

### 1.2 核心功能

- **双模式对比**：同时运行 RAG 模式（使用知识库检索）和 No-RAG 模式（仅基于模型自身知识）
- **多模型支持**：可配置多种大语言模型（如 GPT、Grok、Gemini 等）
- **自动化评估**：使用大模型对输出结果进行自动评分
- **查询重写**：将原始问题转化为更适合 RAG 检索的形式
- **专业领域识别**：自动识别病例所属医学专业领域，限定检索范围
- **患者信息集成**：将患者信息（情境、体格检查、影像学检查）整合到回答上下文中

## 2. 评估标准

系统采用**三维度评估体系**，每个维度评分范围为 **1-10 分**。

### 2.1 医学准确性（权重 40%）

评估回答中医学信息的准确性和可靠性。

| 分数范围 | 评级 | 标准描述 |
|---------|------|---------|
| 10-9 分 | 完全准确 | 所有医学信息符合循证医学标准，无错误 |
| 8-7 分 | 基本准确 | 有少量非关键性偏差，不影响整体医学判断 |
| 6-5 分 | 部分准确 | 存在明显错误，可能影响临床决策 |
| 4-1 分 | 严重错误 | 存在严重医学错误，可能导致医疗风险 |

**评估要点**：
- 诊断依据是否符合医学指南
- 治疗方案是否规范
- 药物使用建议是否准确
- 预后判断是否合理
- 医学术语使用是否正确

### 2.2 关键要点召回率（权重 30%）

评估回答是否涵盖了问题的所有关键要点。

| 分数范围 | 评级 | 标准描述 |
|---------|------|---------|
| 10-9 分 | 完全覆盖 | 涵盖所有关键要点，无遗漏 |
| 8-7 分 | 大部分覆盖 | 覆盖大部分关键要点，有少量非关键遗漏 |
| 6-5 分 | 部分覆盖 | 覆盖部分关键要点，有明显遗漏 |
| 4-1 分 | 严重遗漏 | 严重遗漏关键要点，回答不完整 |

**评估要点**：
- 是否回答了问题的所有子问题
- 是否覆盖了诊断的关键信息
- 是否包含了必要的鉴别诊断
- 是否提供了完整的治疗建议
- 是否考虑了重要的并发症和风险

### 2.3 逻辑完整性（权重 30%）

评估回答的推理过程是否逻辑清晰、结构完整。

| 分数范围 | 评级 | 标准描述 |
|---------|------|---------|
| 10-9 分 | 逻辑完整 | 逻辑链完整，思路清晰，推理严密 |
| 8-7 分 | 基本完整 | 逻辑基本完整，有轻微跳跃但不影响理解 |
| 6-5 分 | 逻辑缺陷 | 逻辑有缺陷，推理过程存在漏洞 |
| 4-1 分 | 逻辑混乱 | 逻辑混乱，无法形成完整的推理链 |

**评估要点**：
- 推理过程是否清晰
- 论据是否支持结论
- 是否存在逻辑跳跃
- 回答结构是否合理
- 因果关系是否明确

### 2.4 综合评分计算

```
综合评分 = 医学准确性 × 40% + 关键要点召回率 × 30% + 逻辑完整性 × 30%
```

## 3. Prompt 设计规范

### 3.1 RAG 模式系统提示词

**文件位置**: `core/processor.py`  
**方法**: `StreamableRAGComparisonProcessor._get_rag_system_prompt()`

```
你是一个专业的医学助手，需要综合患者信息和检索到的医学知识来回答问题。请特别注意并优先使用"知识库搜索"部分提供的信息。

回答要求：
1. 优先参考知识库搜索结果中的信息
2. 将检索到的知识与患者具体情况相结合
3. 严格按照以下格式输出：
```
推理过程：{{基于检索知识和患者信息的推理过程}}
最终结果：{{最终结论}}
```
4. 如果使用了检索到的信息，请在推理过程中明确标注来源，使用`[检索结果 X]`格式
禁止使用 Markdown 语法，请保证逻辑清晰，术语明确，并给出简短、明确的回答
```

**设计原则**：
- **角色定位**：明确模型作为"医学助手"的专业角色
- **优先级指令**：强调优先使用检索到的知识库信息
- **格式约束**：强制使用结构化输出格式（推理过程 + 最终结果）
- **引用规范**：要求标注检索信息来源，便于追溯
- **风格要求**：禁止 Markdown，保证输出简洁清晰

### 3.2 No-RAG 模式系统提示词

**文件位置**: `core/processor.py`  
**方法**: `StreamableRAGComparisonProcessor._get_no_rag_system_prompt()`

```
你是一个专业的医学助手，根据提供的上下文信息和患者信息回答医学相关问题。

请严格按照以下格式输出：
```
推理过程：{{推理过程}}
最终结果：{{最终结果}}
```
如果有引用依据，请在推理过程中明确展示推理依据。
禁止使用 Markdown 语法，请保证逻辑清晰，术语明确，并给出简短、明确的回答
```

**设计原则**：
- **一致性**：保持与 RAG 模式相同的输出格式，便于对比
- **透明性**：要求展示推理依据，提高可解释性
- **简洁性**：强调简短、明确的回答风格

### 3.3 查询重写提示词

**文件位置**: `core/query_rewriter.py`  
**方法**: `QueryRewriter.rewrite_query()`

```
请将以下医学问题转化为更适合知识库检索的查询语句。

原始问题：{original_query}

病例背景信息：
- 情境：{context_info}
- 体格检查：{physical_exam}
- 影像学检查：{imaging}

请返回一个适合向量检索的查询语句，格式：{"rewritten_query": "重写的查询语句"}
```

**设计原则**：
- **上下文增强**：将病例背景信息融入查询，提高检索精度
- **格式约束**：强制 JSON 格式输出，便于程序解析
- **低温度设置**：使用 temperature=0.1，保证输出稳定性

**重写策略**：
- 提取问题中的核心医学概念
- 结合患者症状和检查结果
- 转化为陈述句形式，更适合向量检索
- 保留关键医学术语

### 3.4 医学评估提示词

**文件位置**: `core/evaluator.py`  
**类常量**: `MedicalEvaluator.PROMPT_EVALUATION`

```
你是一个严格的医学评估专家，需要对以下医学回答进行评估。请按照以下三个维度进行评分，每个维度评分范围为 1-10 分：

1. 医学准确性（权重 40%）：
   - 10-9 分：完全准确，所有医学信息符合循证医学标准
   - 8-7 分：基本准确，有少量非关键性偏差
   - 6-5 分：部分准确，存在明显错误
   - 4-1 分：存在严重医学错误

2. 关键要点召回率（权重 30%）：
   - 10-9 分：完全覆盖所有关键要点
   - 8-7 分：覆盖大部分关键要点
   - 6-5 分：覆盖部分关键要点
   - 4-1 分：严重遗漏关键要点

3. 逻辑完整性（权重 30%）：
   - 10-9 分：逻辑链完整，思路清晰
   - 8-7 分：逻辑基本完整，有轻微跳跃
   - 6-5 分：逻辑有缺陷
   - 4-1 分：逻辑混乱

请严格按照以下 JSON 格式输出评分结果：
{
    "medical_accuracy": X,
    "key_point_recall": X,
    "logical_completeness": X,
    "evaluation_reasoning": "详细的评估理由"
}

其中 X 为 1-10 之间的整数。
```

**设计原则**：
- **角色专业化**：定义评估者为"严格的医学评估专家"
- **标准明确**：每个分数段都有清晰的判定标准
- **权重透明**：明确告知各维度的权重
- **格式约束**：强制 JSON 输出，便于程序解析
- **理由要求**：要求提供详细评估理由，提高可解释性

## 4. RAG 查询语句设计

### 4.1 查询重写流程

```
原始问题 → 查询重写模型 → 重写后的查询 → RAG 检索 → 检索结果
           ↑
           └── 融入病例背景信息
```

### 4.2 重写策略

**输入**：
- 原始问题文本
- 患者情境信息
- 体格检查结果
- 影像学检查结果

**输出**：
- 适合向量检索的查询语句（JSON 格式）

**重写原则**：
1. **保留核心概念**：确保关键医学术语不丢失
2. **增加上下文**：将患者症状、检查结果融入查询
3. **转化为陈述句**：陈述句比疑问句更适合向量检索
4. **消除歧义**：明确指代，避免模糊表达

**示例**：

| 原始问题 | 重写后的查询 |
|---------|-------------|
| "患者是否需要手术？" | "脊髓型颈椎病患者的手术指征和手术治疗方案" |
| "这个病怎么治？" | "基于患者症状和影像学检查结果的诊断和治疗方案" |

### 4.3 医学专业领域识别

系统自动识别病例所属的医学专业领域，并在 RAG 检索时限定范围。

**识别方法**：
- 关键词匹配
- 加权计分（长关键词权重更高）
- 选择得分最高的专业领域

**专业领域限定作用**：
- 过滤不相关的检索结果
- 提高检索精度
- 减少噪声干扰

## 5. 医学专业领域分类

### 5.1 专业领域列表

系统支持以下医学专业领域分类：

| 领域代码 | 领域名称 | 关键词示例 |
|---------|---------|-----------|
| `orthopedic_spine` | 骨科脊柱 | 脊髓型颈椎病、椎间盘突出、椎管狭窄 |
| `orthopedics` | 骨科 | 骨折、关节、脊柱、韧带 |
| `neurosurgery` | 神经外科 | 脑部、脊髓、颅内、脑血管 |
| `cardiology` | 心血管科 | 心脏、冠心病、心律、瓣膜 |
| `ophthalmology` | 眼科 | 眼睛、视力、白内障、视网膜 |
| `internal_medicine` | 内科 | 呼吸、消化、内分泌、泌尿 |
| `general_surgery` | 普外科 | 腹部、阑尾、胆囊、甲状腺 |
| `urology` | 泌尿科 | 肾脏、膀胱、前列腺、输尿管 |
| `general` | 通用 | 未匹配到特定领域时使用 |

### 5.2 关键词匹配规则

**权重计算**：
- 长度 ≥ 4 的关键词：权重 × 2
- 长度 < 4 的关键词：权重 × 1

**匹配过程**：
1. 将病例文本转换为小写
2. 遍历各领域的关键词列表
3. 统计每个领域的匹配得分
4. 选择得分最高的领域

### 6.3 并发处理策略

**两级并发**：

1. **案例间并发**：使用 ThreadPoolExecutor 并行处理多个病例
2. **案例内并发**：
   - 模型并发：同时处理多个模型的回答生成
   - 模式并发：同时处理 RAG 和 No-RAG 两种模式
   - 评估并发：同时处理多个评估任务

**线程数配置**：
- 默认线程数：`min(8, 模型数量 × 2)`
- 可在 `config/models_config.json` 中配置 `thread_count`

```json
      "model_id": "google/gemini-2.0-flash-exp:free",
      "api_key_env": "OPEN_ROUTER_KEY",
      "base_url": "https://openrouter.ai/api/v1/"
    }
  ],
  "thread_count": 16

```

**配置项说明**：
- `name`: 模型名称（用于日志和输出标识）
- `model_id`: 模型在 API 提供商处的标识符
- `api_key_env`: 存储 API 密钥的环境变量名
- `base_url`: API 服务端地址
- `thread_count`: 并发线程数

### 7.3 RAG 配置

**文件位置**: `NaiveRag/rag/config.py`

```python
@dataclass
class RAGConfig:
    index_path: Path = Path("NaiveRag/rag_index.faiss")
    meta_path: Path = Path("NaiveRag/rag_meta.json")
    chunk_size: int = 512
    chunk_overlap: int = 50
    top_k: int = 3
    retrieve_k: int = 10
    rerank_enabled: bool = True
    embed_provider: str = "modelscope"
    embed_model: str = "BAAI/bge-m3"
```

**配置项说明**：
- `index_path`: FAISS 索引文件路径
- `meta_path`: 元数据文件路径
- `chunk_size`: 文本切分块大小
- `chunk_overlap`: 切分块重叠大小
- `top_k`: 返回的检索结果数量
- `retrieve_k`: 初始检索结果数量（用于 ReRank）
- `rerank_enabled`: 是否启用 ReRank
- `embed_provider`: 嵌入模型提供商（modelscope/siliconflow/local）
- `embed_model`: 嵌入模型名称

### B. 日志系统

**日志位置**: `logs/` 目录

**日志类型**：
1. **主日志**：带时间戳的操作日志
2. **RAG 查询日志**：记录查询重写和检索结果

**日志级别**：
- DEBUG: 详细调试信息
- INFO: 正常操作信息
- WARNING: 警告信息
- ERROR: 错误信息

### C. 常见问题

#### C.1 RAG 初始化失败
确保已构建 RAG 索引：
```bash
cd NaiveRag
python -m rag.cli --build --md_dir data
```

#### C.2 API 调用失败
检查 `.env` 文件中的 API 密钥配置是否正确。

#### C.3 评估结果为固定值
检查 `core/evaluator.py` 中的 DEBUG 模式是否开启。
