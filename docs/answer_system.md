## 1. 系统概述

### 1.1 系统目标

本系统用于对比评估 RAG（检索增强生成）与 No-RAG（无检索增强）两种模式下，大语言模型在医学问答任务中的表现差异。

### 1.2 核心功能

- **双模式对比**：同一问题分别在 RAG 模式和 No-RAG 模式下生成回答
- **多模型支持**：可同时运行多个大语言模型进行对比
- **自动评估**：使用大模型对生成的回答进行三维度自动评分
- **知识检索**：基于向量数据库检索相关医学知识
- **专业领域识别**：自动识别病例所属医学专科，限定检索范围
- **报告生成**：生成 PDF 回答报告、Excel 评分表、JSON 评估数据

### 1.3 系统特点

- **对比性**：RAG vs No-RAG 的直接对比
- **可解释性**：回答包含推理过程，评估包含理由说明
- **可扩展性**：支持灵活配置模型和评估维度
- **高效性**：多级并发处理，提高处理效率

## 2. 输入与输出

### 2.1 输入数据

#### 2.1.1 病例数据

**格式**: Excel 文件（`.xlsx`）

**内容结构**：

| 列名 | 说明 | 示例 |
|-----|------|------|
| 情境 | 患者主诉、病史等背景信息 | "患者男，65 岁，颈部疼痛伴上肢麻木 3 个月" |
| 体格检查 | 体格检查结果 | "颈活动受限，臂丛神经牵拉试验阳性" |
| 影像学检查 | 影像学检查结果 | "MRI 显示 C5-6 椎间盘突出，脊髓受压" |
| 问题 1 | 第一个问题 | "该患者的诊断是什么？" |
| 问题 2 | 第二个问题（可选） | "需要与哪些疾病鉴别？" |
| 问题 3 | 第三个问题（可选） | "治疗方案是什么？" |

#### 2.1.2 模型配置

**格式**: JSON 文件

**内容**：
- 模型列表（名称、模型 ID、API 密钥、服务地址）
- 并发线程数配置

**调用平台说明**：
- **平台名称**：OpenRouter（模型聚合平台）
- **API 端点**：`https://openrouter.ai/api/v1/`
- **环境变量**：`OPEN_ROUTER_KEY`（存储 OpenRouter API 密钥）
- **示例模型**：`openai/gpt-4o-mini`（通过 OpenRouter 访问的 OpenAI 模型）

**配置示例**：
```json
{
  "models": [
    {
      "name": "GPT",
      "model_id": "openai/gpt-4o-mini",
      "api_key_env": "OPEN_ROUTER_KEY",
      "base_url": "https://openrouter.ai/api/v1/"
    }
  ],
  "thread_count": 16
}
```

**说明**：OpenRouter 是一个模型聚合平台，允许通过统一的 API 访问多种大模型（如 OpenAI、Anthropic、Google 等）。系统通过 OpenRouter 调用大模型 API 进行回答生成和评估。

#### 2.1.3 环境变量

**内容**：
- 各 API 服务的访问密钥

#### 2.2.4 人工评分模板

**位置**: `人工评分.xlsx`

**内容**：
- 供人工评分使用的空白模板

### 3.2 详细处理步骤

#### 步骤 1：系统初始化

**执行内容**：
- 加载模型配置
- 初始化各模型的 API 客户端
- 初始化 RAG 检索引擎（加载 FAISS 索引和元数据）
- 初始化评估器

**前置条件**：
- 模型配置文件存在且格式正确
- RAG 索引已预先构建

#### 步骤 2：读取病例数据

**执行内容**：
- 从 Excel 文件读取病例数据
- 逐行处理每个病例
- 提取患者信息（情境、体格检查、影像学检查）
- 提取问题列表（问题 1、问题 2、问题 3）

**数据验证**：
- 检查必填字段是否存在
- 处理空值和缺失数据

#### 步骤 3：查询重写

**执行内容**：
- 将原始问题文本转化为更适合 RAG 检索的查询语句
- 融入病例背景信息（情境、体格检查、影像学检查）
- 输出 JSON 格式的重写查询

**输入**：
- 原始问题文本
- 患者情境信息
- 体格检查结果
- 影像学检查结果

**输出**：
- 重写后的查询语句

**示例**：
- 原始问题："患者是否需要手术？"
- 重写后："脊髓型颈椎病患者的手术指征和手术治疗方案"

#### 步骤 4：专业领域识别

**执行内容**：
- 分析病例文本（情境 + 体格检查 + 影像学检查 + 问题）
- 使用关键词匹配算法识别医学专业领域
- 选择匹配得分最高的领域

**领域列表**：
- 骨科脊柱、骨科、神经外科、心血管科、眼科、内科、普外科、泌尿科、通用

**匹配规则**：
- 遍历各领域的关键词列表
- 长关键词（≥4 字）权重更高
- 统计各领域匹配得分
- 选择得分最高的领域

#### 步骤 5：RAG 检索

**执行内容**：
- 使用重写后的查询进行向量检索
- 从 FAISS 索引中检索最相关的文档片段
- 使用专业领域关键词过滤检索结果
- 对检索结果进行重排序（ReRank）
- 返回最相关的 Top-K 结果

**检索参数**：
- 初始检索数量（retrieve_k）：10 条
- 最终返回数量（top_k）：3 条
- 重排序：启用

#### 步骤 6：回答生成与评估

此步骤包含多个子流程，详见第 5 章和第 6 章。

## 4. RAG 检索机制

### 4.1 检索流程

```
┌─────────────────────────────────────────────────────────────────┐
│                      RAG 检索流程                                │
└─────────────────────────────────────────────────────────────────┘

  ┌──────────┐
  │ 查询输入  │
  └─────┬────┘
        │
        ▼
  ┌──────────┐
  │ 向量化    │
  │ (Embedding)│
  └─────┬────┘
        │
        ▼
  ┌──────────┐
  │ FAISS    │
  │ 相似度检索 │
  └─────┬────┘
        │
        ▼
  ┌──────────┐
  │ 专业领域  │
  │ 过滤     │
  └─────┬────┘
        │
        ▼
  ┌──────────┐
  │ TF-IDF   │
  │ 重排序   │
  └─────┬────┘
        │
        ▼
  ┌──────────┐
  │ 返回结果  │
  └──────────┘
```

### 4.2 向量化

**嵌入模型**: BAAI/bge-m3

**处理方式**：
- 将查询文本转换为 1024 维向量
- 向量表示查询的语义信息

### 4.3 FAISS 相似度检索

**检索方法**：余弦相似度

**处理过程**：
- 计算查询向量与索引中所有文档向量的相似度
- 返回相似度最高的前 retrieve_k（10）条文档

### 4.4 专业领域过滤

**过滤逻辑**：
- 根据步骤 4 识别的专业领域，获取对应的关键词列表
- 检查检索结果是否包含相关领域关键词
- 保留包含关键词的文档，过滤不相关内容

**目的**：
- 提高检索结果的相关性
- 减少跨领域噪声干扰

### 4.5 TF-IDF 重排序

**重排序逻辑**：
- 结合词汇匹配（TF-IDF）和语义匹配（向量相似度）
- 计算综合得分
- 按综合得分重新排序

**目的**：
- 弥补纯向量检索可能遗漏的关键词匹配文档
- 提高检索精度

### 4.6 结果返回

**返回数量**: top_k（3）条

**返回格式**：
- 文档内容
- 相似度分数
- 重排序分数

## 5. 模型回答生成

### 5.1 双模式生成策略

系统对每个问题生成两种模式的回答：

| 模式 | 说明 | 知识来源 |
|-----|------|---------|
| RAG 模式 | 检索增强生成 | 外部知识库 + 患者信息 |
| No-RAG 模式 | 无检索增强 | 模型内部知识 + 患者信息 |

### 5.2 RAG 模式回答生成

**输入内容**：
- 系统提示词（RAG 模式）
- 问题文本
- 患者信息（情境、体格检查、影像学检查）
- RAG 检索结果

**处理逻辑**：
1. 构建用户请求（包含患者信息和检索结果）
2. 调用大模型 API（通过 OpenRouter 平台：`https://openrouter.ai/api/v1/`）
3. 流式接收模型回答
4. 拼接完整回答

**输出要求**：
- 优先参考知识库检索结果
- 将检索知识与患者具体情况结合
- 标注检索信息来源（使用 `[检索结果 X]` 格式）
- 格式：推理过程 + 最终结果

### 5.3 No-RAG 模式回答生成

**输入内容**：
- 系统提示词（No-RAG 模式）
- 问题文本
- 患者信息（情境、体格检查、影像学检查）

**处理逻辑**：
1. 构建用户请求（仅包含患者信息，无检索结果）
2. 调用大模型 API（通过 OpenRouter 平台：`https://openrouter.ai/api/v1/`）
3. 流式接收模型回答
4. 拼接完整回答

**输出要求**：
- 基于模型自身医学知识
- 展示推理依据
- 格式：推理过程 + 最终结果

### 5.4 多模型并行处理

**处理策略**：
- 对配置的每个模型，同时生成 RAG 和 No-RAG 两种模式的回答
- 使用线程池并发处理多个模型
- 每个模型独立调用，互不干扰

        │

### 6.2 三维度评估

每个回答从三个维度进行评估：

| 维度 | 权重 | 评分范围 | 说明 |
|-----|------|---------|------|
| 医学准确性 | 40% | 1-10 分 | 医学信息的准确性和可靠性 |
| 关键要点召回率 | 30% | 1-10 分 | 对问题关键要点的覆盖程度 |
| 逻辑完整性 | 30% | 1-10 分 | 推理过程的逻辑性和完整性 |

### 6.3 评估请求构建

**输入内容**：
- 病例信息（编号、模型、模式）
- 待评估的回答文本
- 评估提示词（包含评分标准）

**构建方式**：
- 将病例信息和回答文本填入评估提示词模板
- 设置系统提示词（评估专家角色）

### 6.4 评估模型调用

**调用方式**：
- 使用配置的大模型 API（通过 OpenRouter 平台：`https://openrouter.ai/api/v1/`）
- 流式接收评估结果
- 解析 JSON 格式的评分

### 6.5 评分结果解析

**解析内容**：
- 三个维度的分数
- 评估理由

**异常处理**：
- JSON 解析失败时，返回默认分数
- API 调用失败时，记录错误日志

### 6.6 多评估者交叉评估

**策略**：
- 使用多个模型对同一回答进行评估
- 获取多个评估结果
- 可进行评分对比和一致性分析

## 7. 并发处理策略

### 7.1 多级并发架构

系统采用两级并发处理架构：

```
┌─────────────────────────────────────────────────────────────────┐
│                     并发处理架构                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  第一级：病例间并发                                              │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐               │
│  │ 病例 1  │  │ 病例 2  │  │ 病例 3  │  │ ...    │               │
│  └───┬────┘  └───┬────┘  └───┬────┘  └───┬────┘               │
│      │           │           │           │                      │
│      ▼           ▼           ▼           ▼                      │
│  第二级：案例内并发                                              │
│  ┌──────────────────────────────────────────────────┐          │
│  │  ┌────────┐  ┌────────┐  ┌────────┐             │          │
│  │  │ 模型 1  │  │ 模型 2  │  │ 模型 3  │  ...        │          │
│  │  │ RAG    │  │ RAG    │  │ RAG    │             │          │
│  │  │ No-RAG │  │ No-RAG │  │ No-RAG │             │          │
│  │  └────────┘  └────────┘  └────────┘             │          │
│  └──────────────────────────────────────────────────┘          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 第一级：病例间并发

**并发方式**：
- 使用线程池并行处理多个病例
- 每个病例独立处理，互不干扰

**线程数**：
- 根据系统资源动态调整
- 默认与第二级共用线程池

### 7.3 第二级：案例内并发

**并发内容**：
- **模型并发**：同时处理多个模型的回答生成
- **模式并发**：同时处理 RAG 和 No-RAG 两种模式
- **评估并发**：同时处理多个评估任务

**线程数计算**：
- 默认：`min(8, 模型数量 × 2)`
- 可在配置文件中调整

### 7.4 并发优势

- **提高效率**：显著缩短整体处理时间
- **资源利用**：充分利用系统多核 CPU 资源
- **独立处理**：各任务独立，错误不影响其他任务

## 8. 异常处理

### 8.1 异常类型与处理策略

| 异常类型 | 处理方式 | 影响范围 |
|---------|---------|---------|
| API 调用失败 | 记录错误，返回错误信息 | 单个模型回答 |
| RAG 检索失败 | 降级为 No-RAG 模式 | 当前病例的 RAG 回答 |
| 评估解析失败 | 返回默认分数 | 单个评估结果 |
| 文件读写失败 | 记录错误，跳过当前文件 | 单个输出文件 |
| 配置错误 | 终止程序，提示错误信息 | 整个系统 |

### 8.2 错误日志记录

**记录内容**：
- 错误发生时间
- 错误类型和描述
- 错误发生的上下文（病例编号、模型名称等）
- 错误堆栈信息

**日志位置**：
- `logs/` 目录下的日志文件

### 8.3 降级策略

**RAG 降级**：
- 当 RAG 检索失败时，自动降级为 No-RAG 模式
- 记录降级原因
- 继续处理其他任务

**评估降级**：
- 当自动评估失败时，标记为待人工评估
- 不影响回答生成和 PDF 输出

### 8.4 容错机制

**任务隔离**：
- 每个病例、每个模型、每个模式独立处理
- 单个任务失败不影响其他任务

**数据备份**：
- 关键中间结果临时保存
- 支持断点续处理（需手动实现）
